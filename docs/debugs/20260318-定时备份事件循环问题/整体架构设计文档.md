
# 定时备份事件循环问题 - 整体架构设计文档

**创建日期**：2026-03-18  
**问题状态**：设计完成，待实施

---

## 1. 问题回顾

### 1.1 现象
- 手动触发立即备份能够成功备份
- 自动备份计划执行会失败
- 失败耗时非常短（0.01s-0.07s），说明在开始执行后立即失败

### 1.2 错误信息
```
RuntimeError: Task &lt;Task pending ...&gt; got Future &lt;Future pending&gt; attached to a different loop
```

### 1.3 根因分析
1. **FastAPI 应用启动时**：
   - Uvicorn 创建并运行主事件循环
   - SSH 连接池在这个主循环中初始化
   - 连接池的 `asyncio.Lock()` 和异步任务都绑定到主循环

2. **定时备份触发时**：
   - APScheduler 在后台线程中调用 `_execute_backup()`
   - `_execute_backup()` 调用 `asyncio.run()`
   - `asyncio.run()` **创建了一个新的事件循环**

3. **冲突发生时**：
   - 尝试使用已绑定到主循环的 SSH 连接池
   - 导致事件循环不兼容错误

---

## 2. 解决方案概述

采用 **方案1 + 选项1** 的组合：

- **核心修复**：使用 `asyncio.run_coroutine_threadsafe()` 将异步备份任务提交到主事件循环执行
- **辅助优化**：将 hourly 任务按设备 ID 分散到不同分钟执行

## 3. 整体架构设计

### 3.1 修复前的架构（有问题）

```
┌─────────────────────────────────────────────────────────┐
│              FastAPI 主事件循环                          │
│  ┌─────────────────────────────────────────────────┐  │
│  │  SSH 连接池 (asyncio.Lock, 后台任务)             │  │
│  └─────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                          ↑
                          │ 冲突！不同的事件循环
                          ↓
┌─────────────────────────────────────────────────────────┐
│         APScheduler 后台线程 + 新事件循环                │
│  ┌─────────────────────────────────────────────────┐  │
│  │  _execute_backup() → asyncio.run()               │  │
│  └─────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 3.2 修复后的架构（正确）

```
┌─────────────────────────────────────────────────────────┐
│              FastAPI 主事件循环                          │
│  ┌─────────────────────────────────────────────────┐  │
│  │  SSH 连接池 (asyncio.Lock, 后台任务)             │  │
│  ├─────────────────────────────────────────────────┤  │
│  │  execute_scheduled_backup() ←────────────────┐  │  │
│  └─────────────────────────────────────────────────┘  │  │
└─────────────────────────────────────────────────────────┘  │
                          ↑                                   │
                          │ run_coroutine_threadsafe()       │
                          │                                   │
┌─────────────────────────────────────────────────────────┐  │
│         APScheduler 后台线程                             │  │
│  ┌─────────────────────────────────────────────────┐  │  │
│  │  _execute_backup() ─────────────────────────────┘  │  │
│  └─────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────┘
```

## 5. 数据流程

### 5.1 修复前的流程（失败）

```
1. APScheduler 触发定时任务
   ↓
2. 后台线程调用 _execute_backup(device_id)
   ↓
3. asyncio.run() 创建新事件循环
   ↓
4. execute_scheduled_backup() 在新循环中执行
   ↓
5. 尝试使用 SSH 连接池（绑定到主循环）
   ↓
6. ❌ RuntimeError: Future attached to a different loop
```

### 5.2 修复后的流程（成功）

```
1. APScheduler 触发定时任务
   ↓
2. 后台线程调用 _execute_backup(device_id)
   ↓
3. 获取主事件循环引用 (get_main_event_loop())
   ↓
4. asyncio.run_coroutine_threadsafe(coro, main_loop)
   ↓
5. 协程提交到主事件循环队列
   ↓
6. 主循环执行 execute_scheduled_backup()
   ↓
7. 使用 SSH 连接池（绑定到主循环）✅
   ↓
8. future.result() 等待执行完成
   ↓
9. 返回结果
```

## 6. 关键设计决策

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 异步执行方案 | `run_coroutine_threadsafe()` | 改动最小，风险最低，见效最快 |
| 任务分散策略 | 按设备ID取模 | 确定性好，便于调试，分布均匀 |
| 超时时间 | 300秒 | 备份任务可能需要较长时间，5分钟比较合理 |

## 8. 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 主循环引用未初始化 | 低 | 高 | 添加空值检查和日志告警 |
| `run_coroutine_threadsafe()` 异常 | 中 | 高 | 完整的异常捕获和错误处理 |
| 任务执行超时 | 中 | 中 | 设置合理的超时时间（300秒） |
| 任务分散不均匀 | 低 | 低 | 设备ID取模在65台设备下应该均匀 |
