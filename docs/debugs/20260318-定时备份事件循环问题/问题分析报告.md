# 定时备份事件循环问题 - 问题分析报告

**创建日期**：2026-03-18  
**问题状态**：已找到根因

---

## 1. 问题描述

### 1.1 现象
- 手动触发立即备份能够成功备份
- 自动备份计划执行会失败
- 失败耗时非常短（0.01s-0.07s），说明在开始执行后立即失败

### 1.2 环境信息
| 配置项 | 值 |
|--------|-----|
| 生产环境地址 | http://10.23.65.95 |
| 容器名称 | switch_manage_unified |
| 镜像版本 | 20260318-1647 |

---

## 2. 问题分析过程

### 2.1 前端页面验证
通过生产环境前端页面观察到：
- 备份计划页面显示有 65 个备份计划
- 概览页面显示成功率仅为 4-5%
- 最近执行日志显示大量定时备份失败
- 失败的执行耗时极短（0.01s-0.07s）

### 2.2 后端日志分析
查看生产环境后端错误日志（`/var/log/supervisor/backend-error.log`），发现关键错误信息：

```
RuntimeError: Task <Task pending name='Task-514' coro=<BackupService.execute_scheduled_backup() running at /unified-app/app/services/backup_service.py:123> cb=[run_until_complete.<locals>.done_cb()]> got Future <Future pending> attached to a different loop
```

## 3. 问题根因

### 3.1 根本原因
**异步事件循环不兼容问题**

### 3.2 技术细节

#### 问题代码位置
`app/services/backup_scheduler.py` 第 155-157 行：

```python
result = asyncio.run(
    backup_service.execute_scheduled_backup(device_id, db, task_id)
)
```

#### 问题分析

1. **FastAPI 应用启动时**：
   - Uvicorn 创建并运行主事件循环
   - SSH 连接池（`ssh_connection_pool.py`）在这个主循环中初始化
   - 连接池的 `asyncio.Lock()` 和异步任务都绑定到主循环

2. **定时备份触发时**：
   - APScheduler 在后台线程中调用 `_execute_backup()`
   - `_execute_backup()` 调用 `asyncio.run()`
   - `asyncio.run()` **创建了一个新的事件循环**

3. **冲突发生时**：
   - `BackupService.execute_scheduled_backup()` 调用 `netmiko_service.execute_command()`
   - `netmiko_service.execute_command()` 尝试使用 SSH 连接池
   - 连接池中的 `asyncio.Lock()` 绑定在**主事件循环**上
   - 但当前代码运行在**新创建的事件循环**中
   - 导致 `RuntimeError: Task got Future attached to a different loop`

#### 为什么手动备份能成功？
手动备份通过 FastAPI 端点触发：
- 运行在 FastAPI 的主事件循环中
- 与 SSH 连接池使用同一个事件循环
- 没有事件循环冲突问题

| 影响项 | 描述 |
|--------|------|
| 受影响功能 | 定时自动备份 |
| 不受影响功能 | 手动立即备份 |
| 受影响设备 | 所有配置了定时备份计划的设备（65台） |
| 成功率 | 约 4-5%（仅少量设备成功） |

## 6. 临时验证

### 6.1 成功的例外情况
观察到有少量定时备份成功了（如设备"模块33-R04-IPMI接入"），可能的原因：
- 这些设备的连接不在连接池中，需要新建连接
- 新建连接时没有使用已有的绑定到主循环的 Future
