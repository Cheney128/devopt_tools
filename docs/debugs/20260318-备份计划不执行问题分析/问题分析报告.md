
# 备份计划不执行问题分析报告

**分析日期**：2026-03-18  
**问题现象**：备份计划设置为每小时备份，但除了设置计划时备份一次，后续没有自动备份执行

---

## 1. 问题概述

通过分析生产环境和代码，发现备份计划无法按预期自动执行的问题。从前端页面可以看到：
- 多个设备设置了"每小时"备份计划
- 大部分计划显示"从未执行"
- 只有少量计划在创建时有过执行记录

---

## 2. 问题根因分析

通过代码审查，发现了**3个严重问题**：

### 问题1：异步函数与APScheduler不兼容（最严重）

**位置**：`app/services/backup_scheduler.py`

```python
# 第144行
async def _execute_backup(self, device_id: int, db: Session):
    """
    执行设备配置备份
    """
    # ... 异步函数实现 ...
```

**问题描述**：
- `_execute_backup` 被定义为 `async` 函数
- 但 APScheduler 的 `BackgroundScheduler` 是**同步调度器**
- 当调度器触发任务时，它不会等待异步函数执行完成
- 实际上，调度器只是拿到一个协程对象就结束了，任务并没有真正执行

**证据**：查看 `add_job` 调用（第86-93行）：
```python
self.scheduler.add_job(
    func=self._execute_backup,  # 这是一个 async 函数！
    trigger=trigger,
    id=f"backup_{schedule.id}",
    replace_existing=True,
    args=[schedule.device_id, db],  # 传递了 db 会话
    misfire_grace_time=300
)
```

---

### 问题2：数据库会话传递问题

**位置**：`app/services/backup_scheduler.py` 第91行

```python
args=[schedule.device_id, db],  # 问题：传递了当前的 db 会话
```

**问题描述**：
- 在 `add_schedule` 方法中，将当前的 `db` 会话传递给任务
- 但这个 `db` 会话是从 HTTP 请求中获取的
- 当任务在未来（比如1小时后）执行时，这个会话可能已经关闭或失效
- 即使异步问题解决了，数据库操作也会失败

---

### 问题3：启动时备份调度器初始化日志缺失

**位置**：`app/main.py` 第66-68行

```python
try:
    db = next(get_db())
    backup_scheduler.load_schedules(db)
except Exception as e:
    print(f"Warning: Could not load backup schedules from database: {e}")
    print("Application will continue without backup scheduler functionality.")
```

**问题描述**：
- 从容器日志来看，没有看到 "Backup scheduler initialized" 或 "Loading backup schedules from database" 的日志
- 这表明备份调度器可能在启动时就没有正确初始化，或者加载任务时出现了异常但被静默处理了

---

## 3. 代码流程分析

### 当前错误的流程：

```
1. 用户创建备份计划
   ↓
2. 立即执行一次备份（成功，因为是在请求上下文中同步执行）
   ↓
3. 调用 backup_scheduler.add_schedule()
   ↓
4. 将 async 函数 _execute_backup 添加到 APScheduler
   ↓
5. 调度器等待触发时间到达
   ↓
6. 时间到达，调度器调用 _execute_backup()
   ↓
7. ❌ 问题：调度器得到一个协程对象，不等待执行就结束了
   ↓
8. ❌ 没有日志，没有备份执行，任务静默失败
```

---

## 4. 修复方案

### 方案概述：
1. 将 `_execute_backup` 改为同步函数
2. 在任务内部创建新的数据库会话，而不是传递外部会话
3. 使用 `asyncio.run()` 来调用异步的 `collect_config_from_device`

### 详细修复步骤：

#### 修改 `app/services/backup_scheduler.py`：

```python
# 第144行，移除 async 关键字
def _execute_backup(self, device_id: int):
    """
    执行设备配置备份（同步函数）
    """
    import asyncio
    from app.models import SessionLocal
    
    task_id = f"scheduled_{uuid.uuid4().hex[:8]}"
    logger.info(f"Executing backup for device {device_id}, task_id: {task_id}")
    
    started_at = datetime.now()
    execution_log = None
    db = None
    
    try:
        # 创建新的数据库会话
        db = SessionLocal()
        
        # 导入需要的服务
        from app.services.netmiko_service import NetmikoService
        from app.services.git_service import GitService
        
        # 创建服务实例
        netmiko_service = NetmikoService()
        git_service = GitService()
        
        # 调用异步函数，使用 asyncio.run()
        from app.api.endpoints.configurations import collect_config_from_device
        result = asyncio.run(
            collect_config_from_device(device_id, db, netmiko_service, git_service)
        )
        
        # ... 其余逻辑保持不变 ...
        
    except Exception as e:
        # ... 错误处理 ...
    finally:
        if db:
            db.close()
```

#### 修改 `add_schedule` 方法：

```python
# 第86-93行，修改 add_job 调用
self.scheduler.add_job(
    func=self._execute_backup,
    trigger=trigger,
    id=f"backup_{schedule.id}",
    replace_existing=True,
    args=[schedule.device_id],  # 只传递 device_id
    misfire_grace_time=300
)
```

## 5. 验证建议

修复后，建议按以下步骤验证：

1. **检查启动日志**：确认看到 "Backup scheduler initialized" 和 "Loading backup schedules from database"
2. **手动触发测试**：创建一个每分钟执行的备份计划，观察是否按时执行
3. **检查执行日志**：确认看到 "Executing backup for device X" 的日志
4. **查看数据库**：确认 `backup_execution_logs` 表中有新记录

---

## 6. 总结

**根本原因**：
1. APScheduler 是同步调度器，无法正确处理 async 函数
2. 数据库会话在任务执行时可能已失效
3. 任务失败时没有正确的日志记录

**影响范围**：
- 所有定时备份任务都无法正常执行
- 只有创建计划时的手动备份能成功

**修复优先级**：高 - 这是关键功能缺陷
