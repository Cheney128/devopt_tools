
# 备份计划不执行问题 - 设计细化文档

**设计日期**：2026-03-18  
**问题分析文档**：[问题分析报告.md](./问题分析报告.md)

---

## 1. 设计概述

本次设计旨在解决备份计划无法按预期自动执行的问题，同时进行适度的架构重构，提高代码的可维护性和可测试性。

### 1.1 修复的核心问题

1. **异步函数与 APScheduler 不兼容** - `_execute_backup` 是 async 函数，但 APScheduler 的 BackgroundScheduler 是同步调度器
2. **数据库会话传递问题** - 传递了 HTTP 请求中的 db 会话，未来执行时可能已失效
3. **启动日志缺失** - 调度器初始化可能失败但被静默处理

### 1.2 架构优化

将 `collect_config_from_device` 的核心逻辑从 API 端点提取到独立的 `BackupService` 中，实现关注点分离。

---

## 2. 决策记录

| 决策项 | 选择 | 原因 |
|--------|------|------|
| 异步函数处理方案 | 使用 `asyncio.run()` | 改动最小，风险低，不改变现有调度器架构 |
| 数据库会话管理 | 直接使用 `SessionLocal` | 简单直接，完全控制会话生命周期 |
| 日志详细程度 | 详细日志 | 核心功能，出问题时需要快速定位 |
| 重试机制 | 1次简单重试 | 网络抖动等临时性故障较常见，重试1次可显著提高成功率 |
| 架构优化 | 适度重构 | 将核心逻辑提取到 BackupService，API 端点只负责 HTTP 层 |
| 实施策略 | 一并完成 | 修复问题的同时完成架构重构 |

### 4.2 backup_scheduler.py 修改

**主要修改点**：

| 位置 | 修改前 | 修改后 |
|------|--------|--------|
| 第144行 | `async def _execute_backup(self, device_id: int, db: Session)` | `def _execute_backup(self, device_id: int)` |
| 第91行 | `args=[schedule.device_id, db]` | `args=[schedule.device_id]` |
| 内部实现 | 直接调用 `collect_config_from_device` | 创建新 Session → `asyncio.run(BackupService.execute_scheduled_backup())` |
| 错误处理 | 基础异常捕获 | 添加详细日志 + 1次重试（间隔5秒） |

**_execute_backup 实现伪代码**：

```python
def _execute_backup(self, device_id: int):
    import asyncio
    from app.models import SessionLocal
    from app.services.backup_service import BackupService
    
    task_id = f"scheduled_{uuid.uuid4().hex[:8]}"
    logger.info(f"[ScheduledBackup] Starting backup for device {device_id}, task_id={task_id}")
    
    retry_count = 1
    retry_delay = 5
    
    for attempt in range(retry_count + 1):
        db = None
        try:
            logger.info(f"[ScheduledBackup] Creating database session... (attempt {attempt + 1}/{retry_count + 1})")
            db = SessionLocal()
            
            backup_service = BackupService()
            
            logger.info(f"[ScheduledBackup] Calling BackupService.execute_scheduled_backup()...")
            result = asyncio.run(
                backup_service.execute_scheduled_backup(device_id, db, task_id)
            )
            
            if result.get("success"):
                logger.info(f"[ScheduledBackup] Backup completed successfully for device {device_id}")
                return
            else:
                logger.warning(f"[ScheduledBackup] Backup failed for device {device_id}: {result.get('message')}")
                
        except Exception as e:
            error_message = str(e)
            logger.error(f"[ScheduledBackup] Backup failed for device {device_id}: {error_message} (attempt {attempt + 1}/{retry_count + 1})")
            
            if attempt &lt; retry_count:
                logger.info(f"[ScheduledBackup] Retrying in {retry_delay} seconds...")
                import time
                time.sleep(retry_delay)
            else:
                logger.error(f"[ScheduledBackup] Max retries exceeded for device {device_id}")
        finally:
            if db:
                db.close()
                logger.debug(f"[ScheduledBackup] Database session closed")
```

### 4.3 日志设计

**记录的关键日志**：

| 日志级别 | 日志内容 | 说明 |
|---------|---------|------|
| INFO | `[BackupScheduler] Backup scheduler initialized` | 调度器初始化 |
| INFO | `[BackupScheduler] Loading N backup schedules from database` | 加载备份计划 |
| INFO | `[BackupScheduler] Added backup schedule {id} for device {hostname}` | 添加备份计划 |
| INFO | `[ScheduledBackup] Starting backup for device {device_id}, task_id={task_id}` | 开始备份 |
| INFO | `[ScheduledBackup] Querying device info...` | 查询设备信息 |
| INFO | `[ScheduledBackup] Collecting config from device...` | 采集配置 |
| INFO | `[ScheduledBackup] Config changed: {True/False}` | 配置变化判断 |
| INFO | `[ScheduledBackup] Creating execution log...` | 创建执行日志 |
| INFO | `[ScheduledBackup] Backup completed successfully for device {device_id}` | 备份成功 |
| WARNING | `[ScheduledBackup] Backup failed for device {device_id}: {error} (attempt {n}/2)` | 备份失败（将重试） |
| ERROR | `[ScheduledBackup] Max retries exceeded for device {device_id}` | 超过重试次数 |

### 4.4 重试机制设计

**重试策略**：
- 重试次数：1次
- 重试间隔：5秒
- 重试范围：所有异常（简化实现，首次修复不做精细区分）

## 8. 风险评估

| 风险项 | 风险等级 | 应对措施 |
|--------|---------|---------|
| asyncio.run() 在多线程环境下的问题 | 中 | 每个任务创建独立的事件循环，互不干擾 |
| 数据库会话泄漏 | 中 | 使用 try/finally 确保会话关闭 |
| 重构引入新 bug | 中 | 充分测试，保持 API 接口不变 |
