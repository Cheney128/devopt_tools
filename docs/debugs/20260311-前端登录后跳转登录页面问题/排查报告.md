# 前端登录后跳转登录页面问题排查报告

**排查日期**: 2026-03-11  
**问题描述**: 前端登录成功后，点击"用户管理"菜单或刷新页面会跳转到登录页面

---

## 一、问题背景

用户在Build环境（http://10.21.65.20:8081）测试switch-manage应用时，发现以下问题：
1. 登录成功后，点击"用户管理"菜单会跳转到登录页面
2. 刷新页面也会跳转到登录页面
3. 该问题在本地环境也会出现

---

## 二、排查过程与结论

### 事项1：前端axios响应拦截器问题

**排查内容**：
检查 `frontend/src/api/index.js` 中的响应拦截器逻辑

**原始代码**：
```javascript
case 401:
  // 未登录或 Token 过期
  if (config?.url !== '/auth/me') {
    // 只有非初始化请求才显示错误消息和跳转
    ElMessage.error('登录已过期，请重新登录')
    localStorage.removeItem('token')
    if (window.location.pathname !== '/login') {
      window.location.href = '/login'
    }
  }
  break
```

**问题分析**：
- 响应拦截器在收到401错误时，会清除token并跳转登录页
- 排除条件只排除了 `/auth/me` 请求
- 但登录失败（密码错误、验证码错误）也会返回401，此时不应该清除token

**结论**：需要将 `/auth/login` 也加入排除列表

**修复方案**：
```javascript
case 401:
  // 未登录或 Token 过期
  if (config?.url !== '/auth/me' && config?.url !== '/auth/login') {
    // 只有非初始化请求和非登录请求才显示错误消息和跳转
    ElMessage.error('登录已过期，请重新登录')
    localStorage.removeItem('token')
    if (window.location.pathname !== '/login') {
      window.location.href = '/login'
    }
  }
  break
```

### 事项2：验证码验证失败问题

**排查内容**：
检查验证码生成和验证流程

**问题现象**：
- 前端输入的验证码一直提示"验证码错误"
- 从网络请求看，有多个验证码请求被发送

**排查发现**：
```
reqid=114 GET http://localhost:5173/api/v1/auth/captcha [200]
reqid=116 GET http://localhost:8000/api/v1/auth/captcha [200]
reqid=117 GET http://localhost:8000/api/v1/auth/captcha [200]
```

**问题分析**：
- 前端可能在初始化时多次请求验证码
- 每次请求会生成新的验证码ID和验证码
- 前端保存的验证码ID与实际显示的验证码图片不匹配

**结论**：验证码ID与验证码图片存在不匹配的情况，需要检查前端验证码刷新逻辑

---

### 事项3：Token验证失败问题

**排查内容**：
检查JWT Token的生成和验证流程

**问题现象**：
- 登录成功返回Token
- 但调用 `/auth/me` 接口返回401

**排查发现**：
```
Token payload: {'sub': 1, 'exp': 1773201634}
Decoded token: None
```

**问题分析**：
1. Token生成成功，payload正确
2. 但 `decode_access_token()` 返回 `None`，说明解码失败
3. 检查发现 `security.py` 中的 `SECRET_KEY` 获取逻辑存在问题

**关键代码**：
```python
# security.py
SECRET_KEY = getattr(settings, 'SECRET_KEY', None)
if not SECRET_KEY:
    # 如果没有配置，生成一个随机密钥
    SECRET_KEY = secrets.token_urlsafe(32)
```

**问题根因**：
- `getattr(settings, 'SECRET_KEY', None)` 可能返回空字符串或其他falsy值
- 导致每次启动时生成随机SECRET_KEY
- Token用随机密钥签名，验证时用另一个随机密钥，导致验证失败

**结论**：SECRET_KEY配置逻辑存在问题，需要确保使用固定的密钥

尝试重新启动后端服务

**问题现象**：

## 三、问题根因总结

| 问题 | 根因 | 影响 |
|------|------|------|
| 登录后跳转登录页 | axios响应拦截器过于激进地清除token | 任何401错误都会导致登出 |
| 验证码错误 | 验证码ID与图片不匹配 | 无法正常登录 |
| Token验证失败 | SECRET_KEY不一致 | 所有需要认证的接口都无法访问 |

## 八、二次排查（2026-03-11）

### 8.1 问题复现

在本地环境测试时，发现修复后问题仍然存在：
- 登录成功后访问 `/auth/me` 仍然返回 401
- Token payload 中 `sub` 仍然是整数 `1`，而非预期的字符串 `"1"`

### 8.2 深入分析

通过系统调试方法进行排查：

**Step 1: 检查 Token 结构**
```
Payload: {"sub":1,"exp":1773207532}
sub type: <class 'int'>
```

**Step 2: 检查 jose 库行为**
```
Decode token1 (int sub) error: Subject must be a string.
Decode token2 (str sub): {'sub': '1', 'exp': 9999999999}
```

**Step 3: 检查代码逻辑**
直接调用 `create_access_token({'sub': 1})` 返回正确的字符串 sub：
```
Payload: {"sub":"1","exp":1773209673}
```

**Step 4: 检查服务进程**
```
netstat -ano | findstr :8000
  TCP    0.0.0.0:8000    LISTENING    19496
  TCP    0.0.0.0:8000    LISTENING    39860
  TCP    0.0.0.0:8000    LISTENING    39484
```

### 8.3 根因确认

**问题根因**：端口 8000 上存在多个残留的旧进程，这些进程没有加载最新代码，导致：
1. 旧进程使用旧的 `create_access_token` 函数（不转换 sub 为字符串）
2. 新请求被路由到旧进程，返回的 Token 中 sub 仍是整数
3. `jose` 库在解码时验证 `sub` 必须是字符串，导致解码失败

### 8.4 解决方案

**彻底清理旧进程并重启服务**：
```powershell
# 清理所有 Python 进程
Get-Process python -ErrorAction SilentlyContinue | Stop-Process -Force

# 确认端口已释放
netstat -ano | Select-String ":8000"

# 重新启动后端服务
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

# 重新启动前端服务
cd frontend && npm run dev
```

### 8.5 验证结果

清理进程后重新测试：
```
Payload: {"sub":"1","exp":1773210251}
sub type: <class 'str'>

/auth/me Status: 200
SUCCESS! User: admin
```

### 8.6 经验教训

1. **开发环境进程管理**：使用 `--reload` 启动 uvicorn 时，确保旧进程已完全退出
2. **端口冲突检查**：遇到代码修改不生效时，首先检查是否有多个进程监听同一端口
3. **调试优先级**：当单元测试通过但 API 测试失败时，优先检查服务进程状态
