# IP定位 Ver3 整体架构设计

## 1. 背景与目标

当前 IP 定位查询在请求时执行实时关联，存在响应慢、结果受ARP来源设备限制、排障证据不连续的问题。  
Ver3 目标是将“在线动态计算”升级为“离线预计算 + 在线快照查询”，在保持定位准确性的同时显著提升前端体验与可追溯性。

核心目标：
- 查询路径稳定快速，前端列表与搜索优先读取“当前IP定位表”
- 支持“当前/历史”双轨数据，保留审计与回放能力
- 以批次化计算结果固化定位证据，降低重复计算成本
- 为后续跨设备追踪与拓扑增强预留扩展位

## 2. 总体方案

### 2.1 数据分层

系统维护五类核心数据：
- 当前ARP表（`arp_current`）
- 历史ARP表（`arp_history`）
- 当前MAC地址表（`mac_current`）
- 历史MAC地址表（`mac_history`）
- 当前IP定位表（`ip_location_current`）

其中：
- 当前表用于在线查询
- 历史表用于审计、回放、定位争议复核
- 当前IP定位表由离线计算引擎统一刷新

### 2.2 计算链路

定位计算采用固定三段式：
1. `IP -> ARP(MAC, arp_device, vlan, last_seen)`
2. `MAC -> 全网MAC候选(access_device, access_interface, vlan, last_seen)`
3. 候选打分与排序，写入 `ip_location_current`

计算结果同时写入证据字段：
- `match_type`（同设备命中/跨设备命中/ARP-only）
- `arp_source_device_id`
- `mac_hit_device_id`
- `calculated_at`
- `batch_id`

### 2.3 查询链路

前端“IP定位”模块默认只查询 `ip_location_current`：
- 列表分页：按 `confidence DESC, last_seen DESC`
- 搜索定位：按 IP 精确命中或前缀模糊匹配
- 过滤项：位置、是否上联、是否核心设备、可信度区间

在线接口不再做重型关联计算，仅负责读取与过滤。

## 3. 组件划分

### 3.1 采集组件
- 复用现有ARP/MAC采集器
- 采集完成后写当前表并追加历史表
- 生成采集批次号（`collection_batch_id`）

### 3.2 计算组件
- 新增定位计算任务（支持全量/增量）
- 读取当前ARP与当前MAC，生成当前IP定位快照
- 输出计算统计：输入数量、成功数量、降级数量、耗时

### 3.3 查询组件
- API层改为快照读模型
- 保留“实时重算调试接口”作为管理员诊断开关
- 返回字段中包含证据与计算时间，便于前端展示

## 4. 刷新策略

采用“采集触发 + 定时兜底”双机制：
- 采集任务结束后触发增量计算
- 定时任务每日执行一次全量重建（防止漂移）
- 任一计算失败不覆盖旧快照，确保服务连续性

## 5. 一致性与可靠性

- 同一批次写入采用事务与版本号控制
- 快照切换采用“新批次完成后原子替换当前有效版本”
- 失败回滚到上一个有效批次
- 记录完整执行日志，支持定位偏差追踪

## 6. 性能预期

- 查询接口复杂度从“多表动态关联”降为“单表索引读取”
- 常规分页查询响应目标：`P95 < 300ms`
- 计算窗口内前端可持续读取上一版本快照，不阻塞业务

## 7. 范围边界

本期（Ver3）包含：
- 快照数据模型建设
- 离线计算链路建设
- 前端改读快照接口
- 一致性校验与回滚机制

本期不包含：
- 基于LLDP/CDP的多跳拓扑追踪
- 复杂路径可视化渲染
- 自动闭环修复网络配置

## 8. 验收标准

- 前端“IP定位”默认接口不再触发实时重型关联
- 快照表可稳定产出并按批次切换
- 关键接口在目标数据量下满足性能阈值
- 抽样比对“实时重算”与“快照读取”差异可解释且在阈值内

## 9. 数据库自动迁移方案（MySQL）

- Ver3 纳入自动迁移能力，目标数据库仅支持 MySQL（`mysql+pymysql`）
- 启动阶段执行迁移检查：应用启动时先确保迁移历史表，再执行 Ver3 迁移脚本
- 迁移执行要求幂等：重复启动不会重复建表，不会破坏已存在结构
- 迁移失败处理策略：记录失败日志并告警，不覆盖已有可用业务数据
- 非 MySQL 数据库本期不在支持范围，不提供兼容承诺
