# Phase-10-设备角色全链路打通

## 阶段定位

本阶段聚焦“设备角色（`device_role`）”从后端模型到前端设备管理页面的全链路打通，确保字段在数据库、ORM、Schema、API 与前端展示层一致可用。

> 当前状态：✅ 代码实施完成（待验收确认）。

## 背景

在 Phase-8 自动迁移问题修复后，数据库侧 `devices.device_role` 已可落地；但前端“设备管理”页面仍未展示设备角色，后端模型与 Schema 也未完全贯通该字段，导致业务可见性不完整。

## 目标

1. 后端模型恢复并稳定支持 `device_role` 字段读写
2. API 响应包含 `device_role`，并保持向后兼容
3. 前端设备管理页展示“设备角色”列并支持基础筛选
4. 增加回归测试，防止字段回归丢失

## 范围

### 包含范围

- `app/models/models.py` 中 `Device` 模型字段恢复
- `app/schemas/schemas.py` 中 Device 相关 Schema 扩展
- `app/api/endpoints/devices.py` 设备列表与详情响应链路核对
- `frontend/src/views/DeviceManagement.vue` 列展示与筛选入口
- 相关单元测试更新

### 非目标

- 不调整设备角色自动识别策略
- 不新增复杂权限粒度控制
- 不改造历史设备导入流程

## 约束与原则

1. 兼容优先：默认值与空值行为保持可回退
2. 幂等发布：不依赖一次性人工修复步骤
3. 最小变更：仅覆盖 `device_role` 链路，不引入跨模块重构
4. 可验证：每个层级都有对应测试或可观察验证项

## Brainstorming细化结论

### 方案选型（对比后确定）

| 方案 | 描述 | 优点 | 风险 | 结论 |
|------|------|------|------|------|
| A. 仅前端展示补齐 | 只在页面增加列，不改后端契约 | 改动小 | 读写与筛选链路不完整，易回归 | 不采纳 |
| B. 后端优先全链路打通 | 模型→Schema→API→前端→测试逐层贯通 | 一次性解决可见性与可维护性 | 需要跨层联调 | ✅ 采纳 |
| C. 引入新角色服务层抽象 | 新建角色服务统一治理 | 扩展性高 | 超出当前范围，重构成本高 | 不采纳 |

### 分层设计细化

1. **模型层（单一事实字段）**
   - 在 `Device` 模型中稳定映射 `device_role`，与数据库字段保持同名同语义。
   - 允许空值，确保历史数据不因字段缺失导致失败。
   - 索引策略沿用已有数据库结构，不新增跨模块迁移复杂度。

2. **Schema层（接口契约统一）**
   - 在 `DeviceBase/DeviceCreate/DeviceUpdate/Device` 补齐 `device_role`。
   - 保持 snake_case 命名，避免前后端序列化映射分叉。
   - 保证字段为可选，兼容老数据与老调用方。

3. **API层（查询与写入同批验收）**
   - 设备列表、分页、详情接口返回中均包含 `device_role`。
   - 新增/编辑链路接受并回写 `device_role`，避免“只读不可维护”。
   - 筛选接口参数与返回结果语义一致，保证前端筛选可解释。

4. **前端层（可见性与可操作性）**
   - 设备管理表格增加“设备角色”列。
   - 筛选区增加角色选项，并与后端查询参数一一对应。
   - 空值统一展示为“未设置”，避免业务误判为异常值。

5. **发布策略（仅文档阶段）**
   - 当前阶段仅产出方案、测试验证设计与进度同步，不进行代码执行。
   - 执行入口以评审通过为前提，避免未审阅直接开发。
