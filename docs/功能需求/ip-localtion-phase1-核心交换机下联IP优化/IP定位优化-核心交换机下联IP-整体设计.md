# IP地址定位功能优化 - 核心交换机下联IP优化 - 整体设计

## 概述

本文档描述了IP地址定位功能的核心交换机下联IP优化方案，解决核心交换机作为网关设备时，下联接口学习到的大量IP地址导致定位结果不精确的问题。

## 问题描述

在三层网络架构中：

- 核心交换机作为网关设备，会通过ARP学习到全网几乎所有IP地址的ARP表项
- 核心交换机在MAC地址表中通过下联Trunk接口学习到这些MAC地址
- 导致IP定位功能在核心交换机上出现大量不相关的结果，干扰真实位置的判断
- 真实需要的定位结果应该只在接入层交换机上出现

## 解决方案

采用多层次优化方案，结合短期和长期实施方案：

### 短期方案：基于主机名识别核心交换机

- 通过主机名关键词识别核心交换机
- 对核心交换机的接口进行过滤，只保留VLAN接口和特定描述的物理接口

### 长期方案：基于设备角色识别

- 在Device表中新增device\_role字段（核心/汇聚/接入）
- 支持更精确的设备角色管理
- 提供批量设置设备角色的功能

### 核心交换机接口过滤规则

- **保留接口**：VLAN接口 + 描述包含"接入"/"access"等关键词的物理接口
- **过滤接口**：下联Trunk接口、描述包含"上联"/"uplink"等关键词的接口
- **默认策略**：核心交换机上除了明确保留的接口，其他接口默认过滤

### Location字段集成

- 在定位结果中显示设备的location信息
- 支持按机房/位置筛选IP定位结果

### 配置管理

- 创建独立的配置表存储IP定位相关配置
- 支持完全可配置的过滤策略

## 本地核验补充设计（2026-03-23）

基于本地启动前后端人工核验，补充以下设计约束：

1. **优雅降级约束**
   - 当 `ip_location_settings` 表不存在时，配置管理器必须回退默认配置
   - 降级场景应返回可预期业务结果，不直接传播500到页面入口
2. **模型与数据库一致性约束**
   - `Device.device_role` 的使用必须以迁移落地状态为前置条件
   - 迁移未闭环阶段可允许临时保护，但需定义清理时机
3. **自动迁移可靠性约束**
   - 数据库URL特殊字符必须统一编码处理
   - 迁移执行使用受控事务，记录迁移历史
   - 迁移失败不阻断服务启动，但必须暴露告警信息
4. **分阶段闭环策略**
   - 阶段A：可用性稳定（容错保持）
   - 阶段B：结构收敛（字段/表迁移落地）
   - 阶段C：技术债回收（移除临时兼容逻辑）
5. **审查问题修复约束（Phase-9）**
   - 每个任务必须定义：前置依赖、执行边界、量化退出条件
   - 状态语义以 `PROGRESS.md` 为单一事实源，其他文档仅引用不冲突复述
   - API异常输出遵循最小暴露原则，不泄露内部堆栈与敏感配置
   - 迁移失败判定、降级路径与日志等级必须统一口径
   - 文档内容执行脱敏，禁止出现账号、密码、连接串等敏感信息
