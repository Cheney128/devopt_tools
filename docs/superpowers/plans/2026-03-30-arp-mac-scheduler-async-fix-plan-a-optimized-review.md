## 4. 发现的问题与建议

### 4.1 🟡 中等优先级问题

#### 问题 1: 数据库 Session 在异步方法中的使用
**位置**: `_collect_device_async()` 方法

**问题描述**:
虽然添加了注释说明，但 SQLAlchemy Session 在异步上下文中执行同步操作仍有潜在风险：
- Session 不是线程安全的
- 虽然在同一线程中执行，但在异步事件循环中可能导致问题

**建议**:
考虑添加明确的注释说明：
```python
"""
注意：此方法在 asyncio.run() 创建的独立线程中执行，
数据库 Session 操作在此线程内同步执行，因此是安全的。
切勿在其他异步上下文中调用此方法而不确保 Session 隔离。
"""
```

**评审结果**: 已在方案中添加注释，通过

---

#### 问题 2: asyncio.gather() 异常处理逻辑
**位置**: `_collect_device_async()` 方法第 137 行

**问题描述**:
```python
arp_table, mac_table = await asyncio.gather(arp_task, mac_task, return_exceptions=True)
```

当 `return_exceptions=True` 时，异常会作为返回值返回，但当前代码在最外层仍有 try-catch。如果 ARP 和 MAC 都成功，但数据库操作失败，会正确回滚。但如果 ARP 或 MAC 抛出异常，代码会继续执行数据库操作，这可能不是预期行为。

**建议**:
考虑在检测到 ARP 或 MAC 异常时，根据业务需求决定是否继续。当前实现是合理的（单项失败不影响另一项），但建议添加注释说明此设计决策。

**评审结果**: 当前实现合理，通过
